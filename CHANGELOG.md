# Changelog

## [0.1.20] - 2026-05-08

Terminal execution is now always on by default and the "Esecuzione terminale" toggle was removed from the Settings menu. The previous opt-in gate (introduced in 0.1.18 and refined through 0.1.19 with the agent-driven `execution_disabled` reply) added a second layer of safety on top of the existing approval-mode flow, but in practice it was confusing — users would see the agent suggest commands instead of running them and assume something was broken, while the real safeguards (per-command approval card for global scope, ask-mode approval for workspace scope, multi-line command rejection, cwd validation against the workspace folder, foreground timeout, repeated denial guard) were already covering the security envelope. Removed: `WebSocketManager.terminalExecutionEnabled` field plus its `setTerminalExecutionEnabled` / `isTerminalExecutionEnabled` accessors, the `if (!this.terminalExecutionEnabled)` early-return branch in `handleRunTerminalCommand` (the one that returned `status: "execution_disabled"`), the corresponding `selectTerminalExecutionEnabled` host handler and `terminalExecutionUpdate` echo, the third row of the Settings menu, the `terminalExecutionEnabled` field from `getState()` defaults / `syncStateToWebview` / `newChat` / `loadHistory` init payloads, the `state.terminalExecutionEnabled === true` branch of the Reload-Window re-sync block in `restoreState()`, the `applyProposedTerminalExecutionState` per-toggle branching (Run buttons in markdown bash chips are always visible now), and the entire `# TERMINAL EXECUTION GATING` section of `main_agent_prompt.txt` (~45 lines, saving tokens on every turn). The Settings menu drops from four rows to three (Approval / Compressor / Mostra contesto). No migration is needed — leftover `terminalExecutionEnabled` keys in users' webview `vscode.setState` are simply ignored from now on.

## [0.1.19] - 2026-05-07

Settings menu UX cleanup plus two reliability fixes. The approval mode control in the gear menu is now a single "Chiedi approvazioni" toggle (default ON) instead of a segmented Ask/Auto control: the segmented form was inconsistent with the other Settings rows (Egomnia compressor, Esecuzione terminale) which already use plain switches, and the new toggle reads as "ask is the safe default, flip it off if you want full auto for this chat" — same underlying state machine, the message round-trip still posts `selectApprovalMode` with `'ask'` or `'auto'`, so the WS gate, repeated-denial guards, and per-chat persistence all behave exactly as before. The same Settings menu now also exposes a fourth toggle "Mostra finestra di contesto" (default ON) that hides the per-turn metrics row under the composer when the user prefers a cleaner UI; the toggle state is persisted purely in webview `vscode.setState` (like `terminalExpanded[]`) since hiding the row is a visual preference with no runtime side effect — `renderSessionMetrics` short-circuits to `hidden = true` when the flag is off, the cost pill in the chat header keeps rendering independently. The metrics row itself was redesigned from the previous label + progress-bar layout (showing `used / max` of the global context window) into a simpler inline strip showing per-inference data: `Prompt: N token | Output: N token | T secondi`. Token counts come from the existing `metrics.turn.{input_tokens, output_tokens}` payload (per-turn, not session totals); response time is measured purely on the extension side as wall-clock between `setLoading(true)` and `setLoading(false)` and persisted in webview state as `lastTurnElapsedMs`, so no protocol change was needed. The seconds counter is **live**: pressing send resets it to `0.00` immediately (right at the click, not after the IPC round-trip), then a 100ms interval updates only the seconds value-span until the assistant message arrives — when it does, the final elapsed is committed as `lastTurnElapsedMs` and the interval stops. To avoid a visible reset between the immediate webview-side reset and the host's subsequent `loading:true` echo, the loading handler keeps the existing `turnStartedAt` if it was set < 2s ago. Stale latency from the previous chat is cleared on `loadHistory` / `newChat` (rehydrate path) so a freshly opened conversation never displays the previous chat's timing. Settings choices now stick across chats: previously `newChat` and `loadHistory` both called `resetModificationApprovalSession()` which hard-reset the approval mode back to `ask`, wiping the user's explicit Settings choice every time they switched chat; the loadHistory `init` payload also did not propagate the compressor / terminal execution toggles, so the webview rehydrate would coerce them back to false. The reset is now split: `resetMutationDenialState()` keeps clearing the per-task denial guards (those are bound to the active task, not to user intent), while a new `userApprovalPreference` field captures the explicit Settings selection and is reapplied to the runtime mode on every newChat / loadHistory; the `init` payload of loadHistory now also carries `compressorEnabled` and `terminalExecutionEnabled`. The "Approve all for this chat" option from approval cards still flips the runtime mode to `auto` for the current chat only — switching chat reverts to whatever the user picked in Settings — so the question-card label keeps matching its actual scope. Long-running inferences now render the elapsed counter in `1 min e 10 sec` form once it crosses 60 seconds (Italian default; English `1 min 10 sec`, Spanish `1 min y 10 seg` via the new `metrics.turn.minSec` template key in each i18n bundle); below 60s the original numeric format with `secondi` / `seconds` / `segundos` unit is unchanged. Exact-minute values render as `1 min` instead of `1 min e 0 sec` to avoid the dangling zero. The 100ms ticker now updates both the value-span and the unit-span so crossing the 60s threshold mid-inference rerenders cleanly without a flicker. Fixed a regression introduced earlier in this iteration where the live timer would reset mid-turn whenever the host called `syncStateToWebview` (which happens for several legitimate reasons during an inference: system messages, terminal cards, question cards via `moveActiveJobPlaceholderAfterIndex`, agent runtime updates), making the final displayed elapsed smaller than the actual wall-clock time — the rehydrate path used to unconditionally clear `turnStartedAt`, now it preserves the timer when `state.isLoading === true` and only resets when the inference is genuinely over (chat switch / new chat). Terminal cards were redesigned in Claude-Code style: collapsed by default the card now shows only a single quiet line `<status> <semantic title> <chevron>` (e.g. `Eseguito Count and list files in project root ›`) instead of the previous chunky bordered card with `$ command` always visible, scope/mode badges, and rounded status pills. The command, the bash output, the cwd and the exit code all live in an expandable `Bash`-labeled body that opens on click, mirroring how Claude Code's tool-use cards work — meaning conversations with multiple tool calls no longer feel dominated by terminal noise. The semantic title is the agent-provided `explanation` (or the truncated command as fallback). Status text is fully localized: `In esecuzione / Eseguito / Errore in / Annullato / Comando proposto` (it/en/es). The "suggested" state when terminal execution is gated off no longer emits a separate orange chip card — it now uses the same minimal header layout but stays always-expanded with the bash body and a discrete note "Aura non lo eseguirà se non attivi l'esecuzione da terminale". Approval question cards for terminal commands now use the same collapsed-with-chevron layout: header reads "Vorrei eseguire {explanation}" and the actual `$ command`, `cwd`, scope/background metadata are gated behind the chevron, so the user decides on the action description first and only inspects the literal command if they choose to. The legacy `.terminal-suggest-chip*` and the verbose `.terminal-card-status` pill rules were removed; the new design uses VS Code theme tokens (`--vscode-foreground`, `--vscode-descriptionForeground`, `--vscode-errorForeground`, `--vscode-toolbar-hoverBackground`) for theme-coherent typography and respects `prefers-reduced-motion` for the running pulse dot. Backend prompt change (`main_agent_prompt.txt` OUTPUT REQUIREMENTS): the `explanation` argument of tools like `run_terminal_command`, `read_file`, `write_file`, `patch_file`, `convert_document` is now explicitly required to match the user's language — the new terminal-card / approval-card design surfaces those strings as the visible semantic title of the card, so a user typing in Italian must see "Elenca tutti i file nella root del progetto" instead of "List all files in the root of the project". Older messages already in history keep whatever value the agent generated at the time; the rule applies to new tool calls. The terminal-card header was further refined: a small inline terminal icon (1.4px stroke SVG, currentColor) now sits to the left of the status text as a visual anchor for "this is a shell action" — useful when the semantic title alone wouldn't make it obvious. The header font dropped from 12px to 11px so the line reads as low-emphasis metadata to the assistant message that follows, restoring the intended hierarchy where the agent's reply is the primary content and the tool-execution trace is a sidebar annotation. The "Comando proposto" suggested card is gone: when the user has terminal execution turned off in Settings, the extension no longer renders a separate chat card showing the would-be command. The agent now sees the structured `execution_disabled` return and, per the updated `# TERMINAL EXECUTION GATING` prompt section, must reply with a normal conversational assistant message in the user's language explaining it cannot run shell commands right now, then optionally append the suggested command as a fenced ```bash block separated from the prose. Single source of truth for the response; no more duplicate UI. The orphan `concludeTurnAfterSuggestion()` method, the `'suggested'` branch of `handleTerminalCommandEvent` and `renderTerminalCardMessage`, the `terminal-card.suggested` CSS rules, the `term_suggested_*` session-id flow, and the `terminal.status.suggestedSemantic` / `terminal.proposed.execDisabledShort` i18n keys were all removed. The TERMINAL EXECUTION GATING prompt section was further hardened: the agent now MUST produce a single short sentence stating the terminal is off in Settings + at most one fenced ```bash block with the suggested command. Speculation about other causes (empty output, CWD, broken pipes, etc.) is explicitly forbidden; preambles like "Probabilmente…", "Di solito indica…", and closing pleasantries like "Appena mi incolli l'output…" are banned. The earlier wording let the model speculate diagnostics when it saw `status: "execution_disabled"`, producing multi-paragraph error-shaped replies; the new version specifies the cause is known and unique, the reply length is at most 1 sentence + 1 block, and the language must match the user's. Two reliability fixes around the chat "Aura sta lavorando" feedback and final-message rendering. The thinking indicator now stays visible during multi-step jobs that emit intermediate events: previously, every `agentMessage`, `terminalCard`, and `updateMessage` removed the thinking bubble without restoring it, so once the agent ran a terminal command or any tool that produced an in-place card update, the user saw the card sit silently with no progress indicator until the next backend `think_status` arrived (which many models never emit between tool calls). The webview now re-shows the thinking bubble after each of those events when `isLoading` is still true; the question-card flow is intentionally left untouched (it is a blocking interaction, restored by the existing `resumeThinking` after the user answers) and the "suggested" terminal card emitted when terminal execution is gated off correctly does NOT bring back the bubble because the host calls `concludeTurnAfterSuggestion()` to drop `isLoading` first. Second fix: the async job completion path is now deduplicated against the polling fallback. `pollJobStatus` runs every 10s and may race with the WebSocket `job_event completed` (or `failed`/`cancelled`); whichever observed the terminal status first used to clear `currentJobId` via `finishJobSuccess`, which made the early-return guard in `handleJobEvent` ineffective for the second observer, leading to duplicate final assistant messages or duplicate error toasts. `handleJobEvent` now also checks `lastCompletedJobId` and short-circuits if the same job already completed; symmetrically, both code paths now set `lastCompletedJobId` for `failed` / `cancelled` (previously only `completed` was tracked), so the dedupe holds regardless of which terminal event fires.

## [0.1.18] - 2026-04-30

Iteration consolidating four UX and architecture changes around the chat composer and the agent registry. The chat toolbar now ends with a gear-icon Settings menu that gathers three per-chat controls in one place: the approval mode (Ask vs Auto, replacing the standalone "Approvazioni" pill that previously cluttered the toolbar), an Egomnia Compressor toggle that promotes the user-local `~/.aura/skills/Compressor` skill out of the skill picker and into a single switch (the skill is hidden from the picker so it cannot be selected twice, and when on it claims one of the two pinned-skill slots with priority over manual selections), and a new Terminal Execution toggle that gates the `run_terminal_command` tool at the extension layer. When terminal execution is off (the **default** for new chats — explicit user opt-in is required to let the agent run shell commands), the tool returns `{ status: "execution_disabled", suggested_command: ..., explanation: ... }` instead of spawning a process; the agent prompt has a dedicated section instructing the model to surface the command as a fenced ```bash code block with a one-line explanation rather than retrying, and a new orange "Comando suggerito" terminal card with the command + hint is appended to chat as a visual mirror. The proposed-terminal markdown chip in agent replies was redesigned from a heavy two-row card with a "COMANDO SUGGERITO" header into an inline single-line command bar (left accent stripe, `$` prefix in accent color, monospace command, `Copia` + `Esegui` ghost buttons floated top-right). When the terminal toggle is off the `Esegui` button is hidden entirely and the chip carries a tooltip explaining the user's own choice — eliminating the previous contradictory CTA where a green Run button was visible while execution was disabled. The terminal card itself was redesigned in Claude-Code style: by default cards are now collapsed with a one-line `$ <command>` summary plus a chevron, expanding on click to reveal the full body (badges, output streams, exit code, Stop button); the expanded state is persisted per session in webview state and the Stop and chevron buttons stop click propagation so the header-wide click-to-toggle never fights interactive children. Backend-side, the `aura_pdf` and `aura_docx` document agents were removed from the registry and their five tools (`extract_pdf_text`, `extract_pdf_metadata`, `extract_docx_text`, `extract_docx_structure`, `convert_document`) were promoted to the baseline tool set in `AgentDescription` so every agent can read PDF/DOCX directly without a routing handoff; `main_agent_prompt.txt` gained a "DOCUMENT TOOLS" section codifying the PDF/DOCX flow (always start with `extract_pdf_text`, surface `empty_text_layer: true` instead of fabricating content, reject PDF as a `convert_document` target, surface `pandoc_not_installed` cleanly), and the orphan `agent_pdf.txt` / `agent_docx.txt` prompt files were deleted.

## [0.1.17] - 2026-04-30

RAG embeddings provider now follows the LLM the user picks in chat instead of being pinned globally in `.env`. Selecting an Ollama LLM from the model dropdown automatically routes RAG embeddings through the same Ollama host (`bge-m3:latest` by default, `OLLAMA_BASE_URL`); selecting any other provider (OpenAI, Anthropic, generic OpenAI-compatible) keeps RAG on OpenAI embeddings. The new `services/rag_embeddings.py::resolve_rag_embeddings(api_key)` helper is the single source of truth — `BaseRAGManager` no longer reads the global `RAG_EMBEDDING_PROVIDER` env on its own. The env var still works as an explicit override (`ollama` or `openai`) for staging environments that want to pin a provider regardless of api_key, but the per-job dispatch is the default. The Chroma `storage_setting` already includes `(provider, model, base_url)`, so switching provider mid-conversation triggers an automatic index rebuild — no manual cache wipe needed. Avoids the previous footgun where an Ollama-only setup on internal Egomnia hosts would still call out to OpenAI for embeddings.

## [0.1.16] - 2026-04-29

Context-window progress bar in the chat composer now grows monotonically across turns instead of rewinding whenever a new turn happens to be smaller than the previous one. The underlying server-side metric `context.used_tokens` previously reflected only the current turn's tokens, which on providers that don't stream a cumulative `input_tokens` value (Anthropic, several OpenAI-compatible backends) made the bar look stuck or oscillating. It now reflects the cumulative conversation context: live updates during streaming compute `committed.total_tokens + running.turn.total_tokens`, finalization snapshots use the post-commit `committed.total_tokens`, and on backend restart the rehydrate path reconstructs the same cumulative value by summing all jobs' `total_tokens` for the history. Manual context compaction continues to override the value to a smaller post-compact size, so the bar correctly drops after a `/compact`. The server-side database column `agent_jobs.context_used_tokens` is intentionally left storing the per-turn value (no migration needed); cumulative reasoning is centralized in the metrics aggregator only.

## [0.1.15] - 2026-04-29

Approval visibility fix when the Aura chat view is hosted in a non-default container — typically when a developer drags Aura into the Primary Side Bar to live next to (or in place of) the Explorer file tree. In that layout the previous reveal command (`workbench.view.extension.egomnia-aura-sidebar`) targeted an empty container and silently failed, so approval cards were appended to the chat but the view was never brought to the front, making mid-job approvals look broken. The picker now uses the per-view focus command `egomnia-aura.chatView.focus` (which VS Code auto-registers and which works regardless of the hosting container), with the original container command kept as a fallback for older builds, and an explicit `WebviewView.show(true)` call as a final layer. Whenever an approval card is appended, the chat is also force-scrolled to the bottom so the freshly added card never lands below the fold in long conversations. If after the reveal attempt the webview is still not visible (sidebar collapsed, container swapped away), Aura now surfaces a non-modal information toast with a "Show" button that re-runs the reveal flow, so the user is never silently stranded waiting for an invisible approval. Internal `.aura/` metadata writes (workdir id, codebase RAG cache) no longer auto-open in the editor: they were stealing focus during bootstrap or mid-job without any user value, so the auto-open is now skipped for any path that contains a `/.aura/` segment while every real user-facing file write keeps opening as before. The Aura activity bar icon also got a small but meaningful fix: it was previously a colored PNG, which VS Code could not theme automatically, so the Egomnia "e" looked out of place next to the monochrome Files / Search / Source Control icons and worse, when a developer dragged the chat into the Primary Side Bar (next to the Explorer) or the Secondary Side Bar, the colored "e" would float on the activity bar disconnected from any content. The icon is now served from a new `resources/icon-mono.svg` that uses `fill="currentColor"`, so VS Code tints it correctly on light and dark themes and it integrates visually with the rest of the activity bar; the colored marketplace icon stays unchanged. When Aura suggests a shell command in chat as a fenced code block (`\`\`\`bash`, `sh`, `zsh`, `console`, `terminal`, `cmd`, `powershell`, etc.) instead of running it directly, the block is now decorated with a small terminal-icon header labeled "Suggested command" and integrates Copy and Run buttons. Clicking Run hands the command to the same pipeline used by the agent's `run_terminal_command` tool: it spawns under the workspace scope, honors the active approval mode (auto skips the prompt, ask requires confirmation), and streams its output into a real terminal card in chat — exactly as if the agent had executed it. Multi-line snippets keep the Copy affordance but disable Run (the underlying tool refuses multi-line on purpose), so users get a clear hint to paste them in a real terminal instead.

## [0.1.14] - 2026-04-29

UX polish round on the chat composer and toolbar. The “Aura sta elaborando…” / “Aura sta lavorando…” split was unified into a single coherent label across thinking message, progress notification, and job placeholder (it/en/es). The New Chat header button got a chat-bubble icon (with a small inner cross) replacing the bare `+`, making its meaning more immediate. The message composer now persists the in-progress draft across page changes and webview reloads via `vscode.setState` with a 250 ms debounce, restoring the typed text on rehydrate so users no longer lose what they were writing when navigating away. The agent dropdown menu now exposes a per-agent info icon: clicking the inline `i` expands the item showing the agent description, helping users understand the difference between `aura_doc`, `aura_mvp`, `aura_pdf`, etc., directly from the picker. The same info-icon pattern was extended to the skills dropdown: each user skill loaded from `~/.aura/skills` shows a curated, localized one-sentence summary (it/en/es) maintained in `vscode_extension/src/skills/skillSummaries.ts` for the 12 commonly-installed skills (Angular, Compressor, Frontend Pro, Git, Nest, NextJS, React Native, Self Improve, Shadcn, Skill Creator, Spring Boot, UI-UX Pro), with automatic fallback to the first sentence of the SKILL.md frontmatter for custom skills — no ellipsis, no truncation, the sentence always renders in full and matches the active extension language. To make this work even when the picker is at the 2-skills cap, skills are no longer hard-disabled at the DOM level: selection is gated in JS via `aria-disabled`, so the info icon stays clickable on capped skills too. The thinking indicator now persists when the user navigates away from the chat and comes back while a job is still running: `setLoading(true)` is the single point of truth for the “Aura sta lavorando” message and automatically re-shows it after a webview rehydrate, so an in-flight job is never silently invisible after a page change. Hardening: the `thinkingDiv` webview variable was hoisted alongside other top-level webview state to remove a temporal-dead-zone hazard that could otherwise abort the inline-script bootstrap when `setLoading(true)` ran during `restoreState()`, leaving the Aura sidebar visually rendered but completely unclickable; `setLoading` is now wrapped in defensive try/catch boundaries so a future renderer hiccup cannot break the global click model again.
